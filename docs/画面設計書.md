# 画面設計書（CloudOps Change Guard）

> 運用者向け。派手さより「一目でリスクが分かる／危険が埋もれない／承認に必要な情報が1画面で揃う」を優先。**リスクは色だけでなく文字ラベル＋アイコンを併用**（WCAG 2.1 Level A）。production 変更は明確に区別表示。

## 画面一覧

| ID | 画面 | 主な権限 |
|---|---|---|
| SCR-01 | ダッシュボード | 全ロール（範囲はロール調整） |
| SCR-02 | 変更申請一覧 | 全ロール（REQUESTERは自分） |
| SCR-03 | 変更申請作成 | REQUESTER |
| SCR-04 | 変更申請詳細 | 全ロール（権限検証） |
| SCR-05 | 承認待ち一覧 | REVIEWER |
| SCR-06 | ポリシー管理 | ADMIN |
| SCR-00 | ログイン | 公開 |

---

### SCR-01 ダッシュボード
- ステータス別件数（カード）
- 高リスク変更一覧（HIGH/CRITICAL を最上段・アイコン＋ラベル）
- 承認待ち一覧
- 実施予定の変更一覧（scheduled_at 順）
- 最近の監査ログ

### SCR-02 変更申請一覧
- フィルタ：環境 / ステータス / リスク / 申請者
- 各行にリスクレベル（アイコン＋文字＋色）、環境バッジ。**production 行は枠・ラベルで強調**。
- ページネーション。

### SCR-03 変更申請作成
- 入力フォーム（タイトル / 環境 / IaC種別 / 対象サービス / リソース名 / 変更理由 / 変更概要 / 実施予定 / ロールバック手順 / 実施前後チェック雛形）
- 差分テキスト入力
- **リスク判定プレビュー**（preview-risk API。説明文付き findings）
- **ポリシー違反表示**（BLOCK は赤＋「提出できません」明示）

### SCR-04 変更申請詳細（1画面に承認判断材料を集約）
- 変更概要 / 差分（読みやすい表示）
- リスク判定結果（findings：なぜ危険・影響・推奨）
- ポリシー違反（あれば理由）
- 承認履歴（誰が・いつ・決定）
- 実施前チェック（完了状況・本番必須の充足）
- 実施後ヘルスチェック（IaC適用成功とサービス正常性を別表示）
- 監査ログ / コメント
- 操作ボタンは状態とロールに応じて出し分け。**実行開始・却下・ロールバックは確認ダイアログ**。ロールバック手順をすぐ参照可能に。

### SCR-05 承認待ち一覧
- 自分が承認できる変更だけを表示（サーバー側で絞り込み）。

### SCR-06 ポリシー管理（ADMIN）
- 登録済みポリシーの一覧表示（code / 効果 / スコープ / 有効）。MVP は CRUD でなく一覧でよい。

### SCR-00 ログイン
- JWT 認証。エラーは内部情報を出さない汎用文言。

## UX 原則（禁止事項の機械的担保）
- 色のみでリスクを表現しない（文字ラベル・アイコン併用）。
- BLOCK された理由を必ず画面に出す。
- 状態遷移ボタンは許可された遷移のみ活性（不活性理由も提示）。
- production 変更と development 変更を視覚的に区別。

## リスク説明の見せ方（term-board の知見を採用）
- **平易な言葉ファースト**：リスク表示は初期状態で平易な一言（例「世界中の誰でも接続できる状態になります」）を出し、専門用語を未定義のまま連鎖させない。
- **段階的開示**：初期＝平易版 → 「詳しく」で正確な定義（whyDangerous）＋根拠（[リスク判定ルール.md] §6 出典）→ 「なぜ危険？」で深掘り（followUps）。運用者の準備に応じて深さを足す。
- 承認者向けには1行サマリー（recommendedAction）を併記し、1画面で承認判断ができるようにする。
