# 脅威モデル（CloudOps Change Guard）

> 変更管理のライフサイクル（作成 → 承認 → 実施 → 監査）に特有の脅威を列挙し、**サーバー側の緩和策・受入条件・テスト**へ対応づける。認可は必ずバックエンドで強制し、フロントの表示制御は補助に過ぎない（[権限マトリクス.md](./権限マトリクス.md)）。

## 1. 目的と範囲

- **目的**：「申請者・レビュー者・実施者・管理者」が関わる変更統制で、承認を迂回したり痕跡を消したりできないことを、設計と検証で裏付ける。
- **手法**：STRIDE（なりすまし / 改ざん / 否認 / 情報漏えい / サービス妨害 / 権限昇格）を軸に、本アプリ固有の統制脅威（承認バイパス・自己承認・状態遷移の不正・監査改ざん・実施ゲート回避）を重点的に扱う。
- **範囲外**：実 AWS API の実行（抽象化インターフェース越し・記録のみ）、ネットワーク／ホスト層の脅威は [インフラ構成.md](./インフラ構成.md) と [ログ・監視・障害対応設計書.md](./ログ・監視・障害対応設計書.md) 側で扱う。

## 2. 保護対象資産

| 資産 | 守る理由 |
|---|---|
| 変更申請データ | 不正な内容・状態で実施されると本番環境を壊しうる |
| 承認の正当性 | 承認は「人間が責任を持って許可した」証拠。バイパス・自己承認は統制の崩壊 |
| 監査ログの完全性 | 「誰がいつ何をしたか」が改ざん・欠落すると説明責任が果たせない |
| 認証資格情報・トークン | 窃取されると他人になりすませる |
| ポリシー判定結果 | 危険操作の BLOCK が回避されると予防が無力化する |

## 3. 信頼境界

- **ブラウザ ↔ API**：ブラウザ側は信頼しない。フロントの権限出し分けは UX 補助で、**サーバーが最終判定**。
- **API ↔ DB**：永続化は JPA / Prepared Statement のみ。文字列結合 SQL は使わない。
- **外部実行基盤（将来）**：境界の外。実施結果は受け口で**記録するだけ**で、API 自身が apply を実行しない。

## 4. 脅威一覧（脅威 → 緩和策 → 受入条件 → テスト）

凡例（STRIDE）：S=なりすまし / T=改ざん / R=否認 / I=情報漏えい / D=サービス妨害 / E=権限昇格

| ID | 脅威 | STRIDE | 緩和策（実装） | 受入 | テスト |
|---|---|---|---|---|---|
| T-01 | フロントの認可を迂回し API を直接叩いて権限外の操作をする | E | メソッドレベル認可（`@PreAuthorize`）で4ロールをサーバー強制（SEC-2/4）。フロント表示は補助 | A-1 | ChangeRequestAuthorizationTest |
| T-02 | 他人の変更申請を閲覧・編集する（IDOR） | S, I | 所有者検証で不一致は 404（存在秘匿）/ 403（SEC-3） | A-1, A-3 | ChangeRequestAuthorizationTest |
| T-03 | 申請者が自分の申請を自分で承認する（自己承認） | E | 承認時に `approver_id ≠ requester_id` をサーバー検証 | A-2 | ApprovalQuorumTest |
| T-04 | 必要な承認数を満たさないまま実施へ進める（承認バイパス） | E, T | 状態遷移ガードで承認充足を検証＋ポリシーの二重承認（CRITICAL は2名以上） | A-6 | ApprovalQuorumTest, ChangeRequestTransitionTest |
| T-05 | 飛び級・逆行で状態を不正に進める（例 DRAFT→COMPLETED） | T | 許可遷移表のみ受理し、表外は 409 で拒否 | A-9 | ChangeRequestStateMachineTest, ChangeRequestTransitionTest |
| T-06 | 監査ログを改変・削除して痕跡を消す | R, T | 追記専用。UPDATE / DELETE の API・Repository メソッドを設けない | A-8 | AuditLogTest |
| T-07 | 危険操作（本番の削除・置換、全世界公開）を警告のまま通過させる | T, E | サーバー側の遷移ガードで BLOCK し、理由を構造化レスポンスで返す（[リスク判定ルール.md](./リスク判定ルール.md) / [ポリシー一覧.md](./ポリシー一覧.md)） | A-4, A-5 | SubmitRiskGuardTest, RiskEngineTest, RiskAssessmentPersistenceTest |
| T-08 | 本番で必須チェック未完了のまま実施を開始する（安全ゲート回避） | T | production は必須チェック完了を実施開始の前提条件にガード | A-7 | ChangeRequestTransitionTest |
| T-09 | アクセストークンを窃取・再利用する | S | アクセストークンは短命＋リフレッシュをローテーション。401 はインターセプターで一元処理（SEC-7） | — | RefreshTokenRotationTest |
| T-10 | 操作主体を否認する（誰がやったか分からない） | R | すべての遷移と結果記録を監査（操作者・前後状態・要約）。操作種別を網羅的に記録 | A-8 | AuditLogTest |
| T-11 | 差分テキスト等の入力でスクリプトを注入する／存在を露呈させる | I | プレーンテキスト前提で表示時エスケープ（SEC-5）。存在しない／権限外は 404 で秘匿 | — | — |
| T-12 | 入力を介してデータベースを改ざんする（SQL インジェクション） | T | 永続化は JPA / Prepared Statement のみ。文字列結合 SQL を使わない（SEC-6） | — | — |

## 5. 残存リスク・将来対応

- 危険操作（ログイン・承認・実施開始）のレートリミット（SEC-12・SHOULD）、本番 HTTPS 時のセキュリティヘッダ（SEC-13）は将来必要になれば追加する。
- 実施結果の記録は受け口で残すのみで、apply 自体は人間の承認ゲートを前提にする（API は apply を実行しない）。
- 外部通知や実行基盤との連携を増やす場合は、その経路を本表に追記して評価する。
