# インフラ土台 選択ガイド（全アプリ共通ルール）

> 新アプリを作るたびに、**最初にこのガイドで土台(Tier)を1つ選ぶ**。
> 「常に無料」でも「常に AWS」でもなく、**アプリの性質ごとに選択する**のが本ルール。
> 選んだ Tier は `CLAUDE.md` のプロジェクト概要に必ず記録すること。

---

## 0. 判断フロー（上から順に答える）

```
Q1. 他人とデータを共有する？（複数人で同じデータを読み書きする）
     ├─ いいえ ───────────────→ Tier 0（localStorage + 静的配信）
     └─ はい
          │
Q2. 認可が要 or 機微データを扱う？（他人の物を勝手に見せない／編集させない）
     ├─ はい → 認可を仕組みに任せたい？
     │          ├─ はい ──────→ Tier 1（Supabase + R2）★多人数の既定
     │          └─ 自前で書く ─→ Tier 2（Cloudflare 全部入り）
     └─ いいえ（共有はするが認可は軽い）→ Tier 1 か Tier 2
          │
Q3. 1台に複数アプリを相乗りしたい／サーバーを自分で握りたい？
     └─ はい ───────────────→ Tier 3（Oracle Always Free 自前運用）

Q4. AWS の経験を積みたい／特定のマネージドサービスが要る？（意図的な選択）
     └─ はい ───────────────→ Tier 4（AWS）※修練城 防御を全部 ON
```

## 1. Tier 一覧（早見表）

| Tier | 構成 | 料金 | 向くアプリ | 弱点 |
|---|---|---|---|---|
| **0** | localStorage + 静的配信（GitHub Pages 等） | 完全無料・無制限 | 1人用ツール・練習帳・計算機・メモ | 共有不可（自分のブラウザ内だけ） |
| **1** | Supabase（Postgres+認証+RLS）+ Cloudflare R2（画像） | 無料枠 | **多人数で認可が要るアプリ**（レビュー・投稿共有等） | プロジェクト2個上限／DB500MB |
| **2** | Cloudflare 全部入り（Pages+Workers+D1+R2） | 無料枠 | 多人数・自動停止させたくない・複数アプリ | 認可を自前実装／DB は SQLite |
| **3** | Oracle Always Free（常時起動VM 自前運用） | 完全無料（枠形状を守れば） | 複数アプリ相乗り・サーバーを握りたい | 運用負荷・修練城防御が再び必要 |
| **4** | AWS（EC2/RDS/S3 等） | **有料**（無料枠は12ヶ月限定等） | AWS 経験を積む・特定マネージドサービス必須 | 課金事故リスク・修練城 必須 |

### 1-1. 各 Tier の無料枠 上限（現時点の目安・着手時に最新 Pricing を要確認）

#### Tier 0：localStorage + 静的配信（GitHub Pages）
| 項目 | 上限 |
|---|---|
| アプリ数 | **無制限**（リポジトリ単位・無料） |
| localStorage 容量 | 約 5〜10MB／ドメイン／ユーザー（ブラウザ依存） |
| 配信帯域 | 約 100GB／月（ソフト上限） |
| リポジトリ容量 | 約 1GB（推奨上限） |
| ビルド回数 | 10回／時 |
| 料金 | **無料** |

#### Tier 1：Supabase + Cloudflare R2
| 項目（Supabase 無料） | 上限 |
|---|---|
| プロジェクト数 | **2 個** |
| DB（PostgreSQL） | 500MB |
| 認証（MAU） | 50,000／月 |
| Supabase ストレージ | 1GB（※画像は R2 へ逃がす） |
| 転送量（egress） | 5GB／月 |
| Edge Functions | 50万呼び出し／月 |
| 自動停止 | 7日間アクセスゼロで一時停止 |

| 項目（Cloudflare R2 無料） | 上限 |
|---|---|
| ストレージ | 10GB／月 |
| 書込（Class A） | 100万操作／月 |
| 読取（Class B） | 1,000万操作／月 |
| **転送量（egress）** | **無制限（無料）** ← R2 を使う最大の理由 |

#### Tier 2：Cloudflare 全部入り
| 項目 | 上限 |
|---|---|
| Pages リクエスト／帯域 | **無制限** |
| Pages ビルド | 500回／月 |
| Workers リクエスト | 10万／日（アカウント全体で共有） |
| Workers CPU 時間 | 10ms／呼び出し |
| D1（SQLite）容量 | 5GB（アカウント全体で共有） |
| D1 読取／書込 | 読 500万行・書 10万行／日 |
| KV 読取／書込／容量 | 読 10万・書 1,000／日・1GB |
| R2 | 10GB・egress **無制限**（上と同じ） |

#### Tier 3：Oracle Always Free（1台のVM）
| 項目 | 上限 |
|---|---|
| VM（ARM） | 4 OCPU + 24GB RAM（合計・最大4台に分割可） |
| VM（AMD micro） | 2台（各 1/8 OCPU・1GB RAM） |
| ブロックストレージ | 200GB（合計・最大5ボリューム） |
| オブジェクトストレージ | 計 20GB |
| 転送量（egress） | 10TB／月 |
| 自律型DB | 2個 × 20GB |
| 期限 | **無期限**（無料枠の形状を保てば） |
| アプリ数 | VM リソース次第（§3-2：軽量で目安3〜6） |

#### Tier 4：AWS（**有料前提**）
| 項目 | 内容 |
|---|---|
| 12ヶ月 無料枠 | EC2 t2/t3.micro 750h／月・RDS micro 750h／月・S3 5GB 等（**12ヶ月のみ**） |
| 常時 無料枠 | Lambda 100万req／月・DynamoDB 25GB・CloudFront 1TB／月 等 |
| 13ヶ月目以降／枠超過 | **課金（有料）** |
| 料金 | **基本は有料前提**。Budgets 必須（§4） |

> 数値はサービス側でよく変わる。**判断前に各社の公式 Pricing で必ず最新値を確認**すること。

## 2. 無料枠で「先に当たる壁」と対策

| 壁 | 中身 | 対策 |
|---|---|---|
| 画像の**累積保存** | 消さない限り増え続ける | **Cloudflare R2（通信無料・10GB枠）に逃がす** |
| 月の**通信量(egress)** | 利用者×画像表示で増える | 画像を R2 に置けば egress は実質ゼロ |
| **プロジェクト数上限** | Supabase 無料は 2 個まで | 1アプリ1プロジェクトにしない／3個目は別 Tier へ |
| **7日自動停止** | アクセスゼロで眠る | 継続運用アプリは起きない／放置アプリは Tier 0 に寄せる |

> 数値は変わるため、着手時に各サービスの最新 Pricing を必ず一度確認すること。

## 3. 複数アプリ運用の指針

- **1アプリ＝1プロジェクトにしない。** 容量より先に「プロジェクト数」で詰まる。
- **共有が要らないアプリは Tier 0** に寄せる（無料・無制限・枠を消費しない）。
- **画像は必ず R2 へ。** これだけで無料枠の寿命が大きく延びる。
- **「Tier 1 で安全に始め、複数アプリで壁に当たったら Tier 3 へ移行」** を基本線にする。
  アプリ1個も無いうちから10個分を最適化しない。

### 3-1. 【関所】Tier 1（Supabase）を選ぶ前の必須チェック

> 「3個目で壁に当たる」は**忘れる前提**で設計する。下のチェックを `NEW_REPO_CHECKLIST.md` の Tier 選択ステップに組み込み済。**毎回ここを必ず通る**ので、記憶に頼らない。

- [ ] **Supabase ダッシュボードで現在のプロジェクト数を確認**（画面が `N/2` を表示する＝これが唯一の正）
- [ ] 空きがある（0/2 か 1/2）→ Tier 1 を選んでよい
- [ ] **満杯（2/2）→ Tier 1 は選べない**。このアプリは **Tier 2（Cloudflare）か Tier 3（Oracle）** にする
- [ ] どの無料アプリがどの Supabase プロジェクトを使っているかを下の台帳に1行追記

#### Supabase プロジェクト使用台帳（埋まり具合の早見）

| プロジェクト枠 | 使用アプリ | 状態 |
|---|---|---|
| 1 / 2 | （アプリ名を記入） | 空き or 使用中 |
| 2 / 2 | （アプリ名を記入） | 空き or 使用中 |

> 台帳はあくまで「目視の補助」。**最終判断は必ず Supabase ダッシュボードの実数**で行う（台帳はズレ得るが、ダッシュボードはズレない）。

### 3-2. アプリ数が増えたら「1つの Tier に集中させない」

> 「壁が来たら全部 Tier 3（Oracle）へ」は**誤り**。Tier 3 は1台のVMで、10アプリを乗せると別の壁に当たる。
> 多数アプリの正解は**ポートフォリオ全体を散らす**こと。

#### Tier 3（Oracle 1台）を多数アプリの受け皿にできない理由

| 上限 | 10アプリ載せると起きること |
|---|---|
| **RAM 24GB** | 一番効く天井。特に Java/Spring は1個で数百MB〜。10個＋各DBで枯渇 |
| ストレージ 200GB | 画像で圧迫（R2 に逃がせば緩和） |
| **単一障害点(SPOF)** | 1台落ちたら**全アプリ全滅** |
| CPU 共有 | 1アプリの負荷スパイクで**隣も巻き添え**（noisy neighbor） |
| 運用・セキュリティ | 1アプリ侵害で**全VMが影響範囲**。デプロイ・秘密が台数分 |

→ **Tier 3 が向くのは少数（目安3〜6個）の軽量アプリまで。**

#### アプリ数が増えたときの配分指針

| アプリの性質 | 置き場所 | 理由 |
|---|---|---|
| 1人用ツール | **Tier 0（静的）** | 無料・無制限・VMに載せない |
| 共有＋認可が核 | **Tier 1（Supabase）** | RLSで楽。ただし2枠まで（§3-1） |
| **多数の小さな共有アプリ** | **Tier 2（Cloudflare）★多数の本命** | VM不要・SPOFなし・RAM天井なし |
| サーバーを自分で握りたい少数 | **Tier 3（Oracle）** | 3〜6個の軽量まで |

> 原則：**「1つの Tier に集中させない」**。多数アプリの本命は Tier 2（Cloudflare）。
> Tier 3 は「どうしてもサーバーを握りたい少数」用。Java/Spring を自前運用するなら載る数はさらに減る。

### 3-3. アプリ数に「無料の上限」がある（無制限ではない）

> 「10個」は最低限の例。19・29…と増えるほど効くが、**完全無料は多人数アプリの『数』には上限がある**。
> 各無料枠は「アカウント単位の共有プール」を持つため、台数が増えると必ずどこかの天井に当たる。

- **無制限なのは Tier 0（1人用・静的）だけ** — GitHub/Cloudflare Pages はサイト数無制限。1人用は何十個でも無料。
- **多人数アプリは有限** — アカウント単位の共有枠が天井：

  | プロバイダ | アカウント全体で共有する天井 |
  |---|---|
  | Supabase | プロジェクト 2 個 |
  | Cloudflare | Workers 10万req/日・D1 5GB |
  | Oracle | VM 1台・RAM 24GB |
  | GitHub Actions | 2,000分/月 |

- **計画の単位は「アプリ数」でなく「合計負荷」**（DB行数・ストレージ・req・RAM）。数でなく負荷で天井を見る。
- **多人数アプリを増やし続ける戦略：**
  1. **1人用は Tier 0 へ寄せる**（ここは本当に無限）
  2. **プロバイダ口座を分散**（Supabase / Neon / Turso / PlanetScale / Firebase / Cloudflare … 各社の無料枠＝新しいプール）
  3. **自前ホストに相乗り**（Oracle 等の無料VMに軽量アプリを複数）
  4. **ある台数を超えたら小額課金が正直**（完全無料への固執がむしろ非効率になる点が来る）

### 3-4. 試験導入（コミュニティで反応を観る）を想定した設計

> 作ったアプリを**コミュニティ（ディスコード等）で試験導入し、反応＝使われ方・定着を観る**ことを想定する。これには2つの備えが要る。

- **スパイク対策**：リンク共有で**同時アクセスが一気に来る**。小規模でも瞬間的に無料枠のレート／同時接続に当たり得る（Tier 1 の同時接続・egress、Tier 2 の req/日）。
- **観測性（必須・見落としがち）**：「反応を観る」には**計測が要る**が、機能要件に入れ忘れやすい。無料で足せるもの：
  - **Cloudflare Web Analytics（無料・Cookie不要）** … PV・流入
  - **Umami / Plausible（セルフホスト無料）** … イベント・離脱
  - **エラー追跡**（Sentry 無料枠 等）／**簡易フィードバック導線**（フォーム or コミュニティ誘導）
- **試験は使い捨て前提**：反応が無ければ放置 → **Tier 1 の7日自動停止はむしろ好都合**（資源節約）。当たれば本格運用へ。**重い土台を未検証アプリに先行投資しない**。

## 4. AWS（Tier 4）を選ぶときの必須条件

AWS は「既定」ではなく**意図して選ぶ**もの。選ぶ場合は以下を**すべて満たしてから**着手する。

- [ ] 選定理由を `CLAUDE.md` と Issue に明記（用途／必須マネージドサービス名 等）
- [ ] AWS Budgets で予算アラートを設定済み
- [ ] `terraform destroy` / `apply -auto-approve` / `aws *delete*` 系は **AI 単独実行禁止**（CLAUDE.md §8）
- [ ] RDS 等に `lifecycle.prevent_destroy = true` を設定
- [ ] PR ベースのワークフロー（AI が直接 apply しない）
- [ ] バックアップは将来的にクロスアカウント分離を計画

> AWS を選んだ瞬間、課金事故・誤削除のリスクが乗る。**修練城（多層防御）を全部 ON にするのが入場条件**。

## 5. 各 Tier の「できる／できない」対照（共有系アプリの代表機能）

| 機能 | Tier 0 | Tier 1 | Tier 2 | Tier 3/4 |
|---|---|---|---|---|
| 他人の投稿を見る・レビュー | ✕ | ◯ | ◯ | ◯ |
| 本物のログイン・ロール | △見た目のみ | ◯ | ◯ | ◯ |
| 認可（他人の物を守る） | ✕ | ◯ RLS | ◯ 自前実装 | ◯ 自前実装 |
| 画像アップロード | △ブラウザ内 | ◯ R2 | ◯ R2 | ◯ |
| メール通知 | ✕ | △関数追加で可 | ◯ | ◯ |
| 自動停止なし | ◯ | △継続利用なら可 | ◯ | ◯ |
| 試験の観測（分析・計測） | △外部分析のみ | ◯ | ◯ | ◯ |
