# 状態遷移設計（CloudOps Change Guard）

> ステータスは**サーバー側のステートマシン**で強制する。許可遷移のみ受理し、ガード条件を満たさない遷移・許可表にない遷移（飛び級/逆行）は **409 Conflict** で拒否する。フロントの制御は補助に過ぎない。

## 1. ステータス一覧（12種）

| ステータス | 意味 | 種別 |
|---|---|---|
| DRAFT | 下書き（編集可） | 初期 |
| SUBMITTED | 提出済み（レビュー待ち） | |
| UNDER_REVIEW | レビュー中 | |
| APPROVED | 承認済み | |
| REJECTED | 却下 | 終端 |
| RETURNED | 差し戻し（編集可） | |
| SCHEDULED | 実施予定確定 | |
| IN_PROGRESS | 実施中 | |
| COMPLETED | 完了（IaC成功＋正常性確認済み） | 終端 |
| FAILED | 実施失敗 | |
| ROLLED_BACK | ロールバック済み | 終端 |
| CANCELLED | 取消 | 終端 |

---

## 2. 許可遷移表

| From → To | 実行ロール | ガード条件 |
|---|---|---|
| DRAFT → SUBMITTED | REQUESTER(所有者) | 必須項目充足。リスク判定実行。BLOCKポリシー非該当 |
| SUBMITTED → UNDER_REVIEW | REVIEWER または システム | （自動着手可。手動はREVIEWER） |
| UNDER_REVIEW → APPROVED | REVIEWER | 承認要件充足（環境×リスクの承認段数）。自己承認でない。BLOCK違反が残っていない |
| UNDER_REVIEW → REJECTED | REVIEWER | — |
| UNDER_REVIEW → RETURNED | REVIEWER | 差し戻しコメント必須 |
| RETURNED → DRAFT | REQUESTER(所有者) | 差し戻し後、編集のため下書きへ戻す |
| APPROVED → SCHEDULED | OPERATOR | 実施予定日時あり（production×HIGH/CRITICAL は必須） |
| SCHEDULED → IN_PROGRESS | OPERATOR | **production は必須チェック全完了**。BLOCK違反なし。execution 開始記録 |
| IN_PROGRESS → COMPLETED | OPERATOR | iac_apply_result=SUCCESS かつ service_health_confirmed=true |
| IN_PROGRESS → FAILED | OPERATOR | iac_apply_result=FAILED または正常性異常 |
| FAILED → ROLLED_BACK | OPERATOR | rollback_performed=true・rollback_note 記録 |
| {DRAFT, SUBMITTED, UNDER_REVIEW, APPROVED, SCHEDULED} → CANCELLED | 所有者 / ADMIN（実施系はOPERATOR/ADMIN） | 取消理由を記録 |

**終端**：REJECTED / COMPLETED / ROLLED_BACK / CANCELLED からは遷移しない。
**差し戻しの流れ**：UNDER_REVIEW → RETURNED → DRAFT（編集）→ SUBMITTED で再提出。RETURNED から直接 SUBMITTED へは進めない（必ず DRAFT を経由し編集・再リスク判定を通す）。

---

## 3. 遷移図（テキスト）

```
DRAFT ──submit──> SUBMITTED ──review──> UNDER_REVIEW ──approve──> APPROVED ──schedule──> SCHEDULED ──start──> IN_PROGRESS ──┬─complete─> COMPLETED*
  ▲                                          │  │                                                                           └─fail────> FAILED ──rollback─> ROLLED_BACK*
  │                                          │  └─return─> RETURNED ──(edit)──> DRAFT
  └───────────────────────────────────────  │  └─reject─> REJECTED*
   {DRAFT,SUBMITTED,UNDER_REVIEW,APPROVED,SCHEDULED} ──cancel──> CANCELLED*

* = 終端。cancel は所有者/権限者（実施系は OPERATOR/ADMIN）のみ。
```

---

## 4. ガード条件の詳細（サーバー検証）

- **編集可能なのは DRAFT / RETURNED のみ**、かつ所有者（requester_id == 認証ユーザー）。APPROVED 以降は申請者でも編集不可（受入 A-3）。
- **SUBMIT 時**：必須項目（title / target_environment / iac_type / target_aws_service / target_resource_name / change_reason / change_summary / diff_text）を検証。リスク判定を実行し、BLOCK ポリシー該当なら遷移を拒否し理由を返す（受入 A-4/A-5）。
- **APPROVE 時**：環境×リスクの承認段数を満たすまで APPROVED にしない。
  - development×LOW：承認なし（または簡易承認1）で APPROVED 可
  - staging×MEDIUM：REVIEWER 1名
  - production×HIGH：REVIEWER 1名＋実施予定日時＋ロールバック手順
  - production×CRITICAL：REVIEWER 2名以上＋実施予定日時＋ロールバック手順＋実施前チェック完了（受入 A-6）
  - 全ケースで approver_id ≠ requester_id（受入 A-2）
- **START（→IN_PROGRESS）時**：production は is_required な pre_execution_checks が全 is_completed=true でなければ 409（受入 A-7）。
- **COMPLETE 時**：execution に iac_apply_result=SUCCESS かつ service_health_confirmed=true がある場合のみ。IaC成功だけでは COMPLETED にしない（受入 A-10）。
- **すべての遷移で監査ログを before_status/after_status 付きで記録**（受入 A-8）。

---

## 5. 不正遷移時のエラー応答（形式統一）

不正遷移・ガード未充足は 409 Conflict。レスポンスに**エラーコード／ユーザー向けメッセージ／詳細理由／現在の状態／許可される操作**を含める。

```json
{
  "error": {
    "code": "ILLEGAL_STATE_TRANSITION",
    "message": "この変更申請は現在の状態では完了にできません",
    "reason": "DRAFT から COMPLETED へは遷移できません。提出・レビュー・承認・実行を経る必要があります。",
    "currentStatus": "DRAFT",
    "allowedActions": ["submit", "cancel"]
  }
}
```

`allowedActions` は現在状態・ロールから算出した、その利用者が実行可能な遷移イベント一覧。

## 5.5 承認フロー・チェック実データ（実装セッションで使用）

- `backend/src/main/resources/data/approval-flow.json`：環境×リスクの承認要件マトリクス（全12セル）＋不変条件（自己承認禁止・承認後編集ロック・本番は必須チェック完了まで実行開始不可・BLOCK違反は遷移不可）。approve/start ガードが参照。
- `backend/src/main/resources/data/checklist-defaults.json`：実施前チェック標準7項目（本番必須）＋実施後ヘルスチェック8項目＋結果値（HEALTHY/WARNING/UNHEALTHY/NOT_CHECKED）＋COMPLETED 条件（IaC成功かつサービス正常）。

## 6. 実装方針

- `ChangeRequestStatus` enum と `ChangeRequestStateMachine`（許可遷移マップ＋ガード関数）を Domain Service 層に置く。Enum だけでなく**遷移ルールを明示的に実装**する。
- 遷移は `transition(cr, event, actor)` 経由のみ。Controller から status を直接 set しない（受入 A-9 / 禁止事項「自由に直接変更できる設計にしない」）。Controller に業務ロジックを集中させない。
- 不正遷移・ガード未充足は `IllegalStateTransitionException`（currentStatus・allowedActions を保持）→ グローバル例外ハンドラで 409＋上記 JSON。
- すべての遷移は監査ログ（before/after ステータス・操作種別・要約）を記録。
