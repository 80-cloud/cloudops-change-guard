# リスク判定ルール（CloudOps Change Guard）

> リスク判定は**単純な文字列一致で終わらせない**。差分テキストを正規化（リソース型・アクション・主要プロパティを抽出）し、構造化項目（対象環境・対象サービス・リソース名）と**組み合わせて**評価する。

> **知見ベース**：各ルールの「なぜ危険か／想定影響／推奨対応」は、AWS公式ドキュメントおよび一次運用知見を体系化した社内ナレッジベース（AWS図鑑：公開 `80-cloud/aws-zukan`）の横断ルール・サービス本番前提に基づく。出典の対応は §6。形容詞でなく**根拠のある説明文**を返すための土台。

## 1. 判定の流れ

```
diff_text + 構造化項目
   │  ① 正規化（IaC種別ごと）
   ▼
正規化された変更 [{ resourceType, action(create/update/delete/replace), properties{}, environment }]
   │  ② 各ルールの述語で評価
   ▼
findings[] （ルールごとに level / why / impact / recommended / isBlock / requiresAdditionalApproval）
   │  ③ 集約
   ▼
RiskAssessment { riskLevel = max(findings.level), isBlocked = any(isBlock), requiresAdditionalApproval = any(...) }
```

- **正規化**：
  - Terraform：`# ... will be destroyed` / `-/+ destroy and then create replacement` / `resource "aws_db_instance"` 等のプラン構文からアクションとリソース型を判定。
  - CloudFormation：変更セット相当のテキストから `Action: Remove/Replace`・リソースタイプ（`AWS::RDS::DBInstance` 等）を判定。
  - 抽出器は **IaC種別ごとのパーサ実装（インターフェース `IacDiffParser`）** に分離し交換可能にする。
- リスクレベル：`LOW < MEDIUM < HIGH < CRITICAL`。最終 risk_level は findings の最大値。

## 2. リスクレベル定義

| レベル | 目安 |
|---|---|
| LOW | 影響軽微・容易に戻せる（タグ追加、出力変更等） |
| MEDIUM | 影響あり・要レビュー（設定変更、容量変更等） |
| HIGH | 重大・本番影響・要強い統制 |
| CRITICAL | 不可逆・全断/重大セキュリティ・二重承認やブロック対象 |

## 3. 検知ルール一覧（最低13件）

| code | ルール名 | 検知条件（正規化後） | level | isBlock | 追加承認 |
|---|---|---|---|---|---|
| RDS_DELETE_OR_REPLACE | RDS削除/置換 | RDS が delete または replace | CRITICAL | prod:✅ | ✅ |
| S3_BUCKET_DELETE | S3バケット削除 | S3 bucket が delete | HIGH(prod:CRITICAL) | prod:✅ | prod:✅ |
| SG_OPEN_WORLD | SG 0.0.0.0/0 許可 | SecurityGroup ingress に 0.0.0.0/0 | HIGH | ✅ | – |
| SSH_22_OPEN_WORLD | SSH22 全世界公開 | ingress port 22 かつ 0.0.0.0/0 | CRITICAL | ✅ | – |
| IAM_ADMIN_ACCESS | IAM管理者権限付与 | AdministratorAccess アタッチ/付与 | CRITICAL | prod:✅ | ✅ |
| IAM_WILDCARD | IAM過剰権限 | policy に Action:* または Resource:* | HIGH | – | ✅ |
| CLOUDTRAIL_STOP_DELETE | CloudTrail停止/削除 | CloudTrail delete/停止 | CRITICAL | ✅ | ✅ |
| AWSCONFIG_STOP_DELETE | Config停止/削除 | AWS Config delete/停止 | HIGH | – | ✅ |
| CW_ALARM_DELETE | CloudWatch Alarm削除 | Alarm delete | MEDIUM | – | – (理由必須はポリシー側) |
| ALB_DELETE | ALB削除 | LoadBalancer(ALB) delete | HIGH | – | ✅ |
| TARGET_GROUP_DELETE | Target Group削除 | TargetGroup delete | HIGH | – | ✅ |

| PROD_ECS_LARGE_CHANGE | 本番ECS大規模変更 | production かつ ECS service の desiredCount/taskDef 大幅変更 | HIGH | – | ✅ |
| TF_DESTROY_BULK | Terraform destroy相当の一括削除 | plan に複数リソースの destroy（`will be destroyed` 多数 / `terraform destroy`） | CRITICAL | prod:✅ | ✅ |

> 上表の level/isBlock は既定。最終的な BLOCK・追加承認・理由必須・メンテ時間必須は **ポリシーエンジン**（[ポリシー一覧.md]）が環境とあわせて確定する（リスク判定＝危険の説明、ポリシー＝統制の強制、と責務分離）。

## 4. finding の出力（人間が理解できる説明文）

各 finding は必ず次を返す（赤黄緑だけにしない）：

```json
{
  "ruleCode": "SSH_22_OPEN_WORLD",
  "ruleName": "SSHポート22の全世界公開",
  "riskLevel": "CRITICAL",
  "whyDangerous": "SSH(22)を0.0.0.0/0へ開放すると、インターネット全体から総当たり攻撃の対象になります。",
  "expectedImpact": "踏み台化・不正ログイン・横展開による本番侵害の恐れ。",
  "recommendedAction": "送信元を社内IP/VPN/SSMセッションマネージャに限定し、22の全開放を取り消してください。",
  "isBlock": true,
  "requiresAdditionalApproval": false
}
```

## 4.5 ルール実データ（実装セッションで使用）

説明文の実データを `backend/src/main/resources/data/risk-rules.json`（14ルール）に作成済み。RiskEngine はこのデータを読み、各 `RiskRule` 実装が検知ロジックを担い、説明文・統制属性はデータ駆動にする（ADR-0003 ルールベースに沿う）。

各エントリのスキーマ：
```
ruleCode, ruleName, baseRiskLevel, envEscalation?,
detection { summary, terraformHints[], cloudformationHints[] },
whyDangerous(正確な定義/条件つき), plainMeaning(平易版), expectedImpact(場面),
followUps[ {q,a} ], recommendedAction(一言), approverSummary(承認者向け1行),
control { isBlock?/isBlockInProd?, requiresAdditionalApproval, requiresReason, requiresMaintenanceWindow },
awsZukanRefs { commonRules[], services[] }, sources[]
```
> `whyDangerous` は ADR-0004 に従い**条件つき**（「削除保護が無い場合は…」）で記述。`sources` は AWS公式 / aws-zukan の出典。

## 5. 設計・テスト方針

- ルールは `RiskRule` インターフェースの実装として1ルール1クラス（M-3）。`RiskEngine` が全ルールを評価し集約。
- **各ルールに単体テスト**（該当差分で検知、非該当で不検知、環境による level 変化）。受入 A-4/A-5 をカバー。
- 誤検知/見逃しは R-01 のリスク。サンプル差分のフィクスチャを用意し回帰テスト化。

## 6. 各ルールの根拠（AWS知見ベース／aws-zukan 連携）

> 出典＝AWS図鑑（公開 `80-cloud/aws-zukan`）の横断ルール（common-rules）・サービス本番前提（productionPrereqs）・一次運用知見（realWorldNotes：公開Qiita / AWS公式）。リスク説明文（whyDangerous / expectedImpact / recommendedAction）はこの根拠から起こす。

| ruleCode | aws-zukan 根拠（横断ルール / サービス本番前提・知見） |
|---|---|
| RDS_DELETE_OR_REPLACE | `backup`（自動バックアップ・別アカウント複製・復元テスト）／ service `rds` 本番前提=private配置/SG設計/自動バックアップ／`deploy-audit`⑤⑥（削除/置換の目視・削除保護） |
| S3_BUCKET_DELETE | `encryption`・`iac-state`（state置き場はバージョニング/暗号化/パブリックアクセス遮断、機密が平文で入りうる）／ service `s3` 本番前提 |
| SG_OPEN_WORLD | `network`（許可は最小・許可元はIPよりSG優先）／ service `securitygroup` 本番前提=最小許可 |
| SSH_22_OPEN_WORLD | service `securitygroup` 知見：「SSH(22)を0.0.0.0/0で開けると総当たりの標的。自分のIPに絞る／Session Manager／ALB経由でSSH不要なら開けない」／`network` |
| IAM_ADMIN_ACCESS | `iam`（最小権限・長期キーを機械に置かない）／`iam-templates`（役割分離）／ service `iam` 本番前提 |
| IAM_WILDCARD | `iam`（最小権限）／`deploy-audit`（権限の点検） |
| CLOUDTRAIL_STOP_DELETE | service `cloudtrail` 本番前提=全リージョン証跡/ログ用S3保護(改ざん防止)/保持設計／`logging`（改ざんされにくい保管・専用アカウント） |
| AWSCONFIG_STOP_DELETE | service `config`（構成変更の継続記録・準拠評価）／`logging` |
| CW_ALARM_DELETE | service `cloudwatch` 知見：「アラーム編集で通知/自動復旧アクションが片方消えることがある」→削除は監視喪失に直結／`monitoring`（三層監視・しきい値検知） |
| ALB_DELETE | service `alb` 本番前提=最低2AZ/各AZにpublicサブネット/ターゲットグループ／`monitoring` |
| TARGET_GROUP_DELETE | service `alb` 知見：「TGがVPC/ターゲット種別と噛み合わないと転送先に出ない」→削除はルーティング喪失 |
| PROD_EC2_REPLACE | service `ec2` 本番前提=IAMロール/監視/バックアップ/パッチ／`deploy-audit`／知見=EIP/パブリックIPv4課金 |
| PROD_ECS_LARGE_CHANGE | service `ecs` 知見：「healthCheckGracePeriodSeconds 既定0で起動前にunhealthy→再起動ループ」／本番前提=タスク実行ロール最小権限/ALBヘルスチェック/Logs出力 |
| TF_DESTROY_BULK | `deploy-audit`⑤⑥（plan差分の意図しない削除/置換を目視・削除保護で既定ワンコマンド削除を防ぐ）／`iac-state`（同時applyロック） |

> **設計思想との一致**：aws-zukan の `deploy-audit`（apply前点検：①機密grep ②環境固有値の外部化 ③無料枠/予算 ④版数突合 ⑤plan差分の削除/置換目視 ⑥削除保護）は、本アプリの「変更前に危険を見つける」中核そのもの。リスク判定＋ポリシー＋実施前チェックは、この点検を**人手でなく仕組みで強制**する実装である。

## 7. 説明文の設計原則（既存アプリの知見から）

リスク判定結果（whyDangerous / expectedImpact / recommendedAction）の**書き方**は、既存アプリの設計判断を採り入れる。

### 7-1. ルールベースで生成し、文章自動生成に頼らない（aws-zukan ADR-0003）
- 説明文は**ルールごとに用意した定型から組み立てる**。LLM の自由生成は必須にしない（正確性・説明可能性・再現性を優先）。ユーザー要件「AIの自由生成は必須でない／正確で説明可能な要約」と一致。

### 7-2. 断定せず前提条件つきで示す（aws-zukan ADR-0004）
- 「削除は常に危険」と断定せず、**条件つき**で示す：「**削除保護が無い場合は**データ消失リスクが高い」「**バックアップ未確認なら**復旧不能の恐れ」。誤検知（R-01）を減らし、運用者の納得感を上げる。

### 7-3. 説明の5層モデル（term-board の用語提示構造）
term-board が用語を「正確な定義／平易版／現場の場面／深掘り／一言」で提示する5層構造を、リスク説明に採り入れる：

| 層 | リスク説明での対応 | 例 |
|---|---|---|
| 正確な定義（meaning） | whyDangerous（技術的に正確） | 「SG ingress に 0.0.0.0/0 を許可する変更」 |
| 平易版（plainMeaning） | 運用者向けの一言 | 「世界中の誰でも接続できる状態になります」 |
| 現場の場面（scene） | expectedImpact | 「踏み台化・総当たり攻撃の起点になりうる」 |
| 深掘り（followUps） | なぜ危険／対策 | 「なぜ？→攻撃面が広がる。対策→送信元をSG/自IPに限定」 |
| 一言（recommendedAction） | 推奨対応＋承認者向け要約 | 「0.0.0.0/0 開放を取り消し、Session Manager 等に切替」 |

> 画面では**平易な言葉ファースト＋段階的開示**（[画面設計書.md]）で提示し、初期は平易版、「詳しく」で正確な定義・根拠を開く。
