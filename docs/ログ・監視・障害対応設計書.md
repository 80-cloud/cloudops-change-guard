# ログ・監視・障害対応設計書（CloudOps Change Guard）

> 監視（既知の問題を見張る）とオブザーバビリティ（未知も後から調査できる）を区別する。本アプリは「変更後に正常性を確認し、失敗時にも復旧できる状態を残す」ことが価値の中核であり、ログ・監査・ヘルスチェックはその実体である。

## 1. ログ設計

- **構造化ログ**（JSON）で出力し機械解析可能にする。
- `requestId` / `userId` を MDC で自動付与し追跡性を上げる。
- ログレベルは dev/prod プロファイルで強度を分ける。
- **監査ログ（audit_logs）はアプリ機能としての追記専用記録**であり、運用ログ（標準出力）とは別物。監査ログは改ざん不可（[ER図.md] 2-12）。

## 2. 監視（4ゴールデンシグナル）

| シグナル | 本アプリでの観点 |
|---|---|
| Latency | API 応答（一覧・詳細・リスク判定）p95 |
| Traffic | リクエスト数・申請件数の推移 |
| Errors | 4xx/5xx 率・ポリシー違反検知数・状態遷移拒否数 |
| Saturation | DB 接続・コネクションプール使用率 |

> MVP は最小限（アプリログ＋ヘルスエンドポイント）。本番運用時に Prometheus/CloudWatch 等へ拡張（将来）。

## 3. アプリ内「実施後ヘルスチェック」との関係

本アプリの **実施後ヘルスチェック（post_execution_health_checks）** は、変更対象の AWS 環境側の正常性を運用者が記録する機能であり、アプリ自身の監視とは別レイヤ。

- 記録項目：IaC適用結果 / ALB Target Health / EC2・SSM 状態 / HTTPヘルス / CloudWatch Alarm / アプリ疎通 / DB接続 / 備考。
- 結果区分：正常(OK) / 警告(WARN) / 異常(ERROR) / 未確認(UNKNOWN)。
- **「IaC適用成功」と「サービス正常性確認済み」を区別**し、両者が揃って初めて COMPLETED とする。

## 4. 障害対応（C→A：なぜなぜ分析）

```
① 異常検知 → ② 影響確認・切り分け → ③ 一次対応（止血）
   → ④ 原因特定（症状と原因を分離・再現） → ⑤ 根本修正 → ⑥ 再発防止
```

- 変更起因の障害は、当該 change_request の実施後ヘルスチェック＝ERROR と FAILED 遷移で記録。
- 復旧は ROLLED_BACK 遷移＋rollback_note で痕跡を残す（失敗時にも復旧できる状態を残す）。
- 再発防止は本書または Issue に表で残す（問題→対策）。

## 5. Runbook（最小）

| 事象 | 一次対応 |
|---|---|
| 変更後にサービス異常（ヘルスチェック ERROR） | ロールバック手順を参照し ROLLED_BACK 記録。関係者連絡 |
| ポリシー BLOCK で提出不可 | 違反理由を確認し、設計見直し or 正規承認プロセスへ |
| 状態遷移拒否（409） | 許可遷移表とガード条件（[状態遷移設計.md]）を確認 |
