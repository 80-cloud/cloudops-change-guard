# ADR-0001: 変更申請の状態遷移をサーバー側で一元管理する

- ステータス: 採用
- 日付: 2026-06-30

## 背景
変更申請は 下書き→提出→査閲→承認→実施→完了 と状態が進む。
状態の判定を画面や各APIに散らすと、飛び級・逆行・終端からの再操作といった
不正な遷移を見落としやすい。

## 決定
許可する遷移（現在状態×操作→次状態・許可ロール・前提条件）を1つの表に集約し、
サーバー側のステートマシンだけが状態を変える。コントローラやサービスが状態を
直接書き換えることは禁止する。表に無い操作は 409 Conflict で拒否し、現在状態と
許可操作を返す。

## 検討した代替案
- 画面側で遷移可否を制御する: UI を変えれば回避でき、信頼できない。
- 各APIで個別にチェックする: 重複と抜けが生じ、全体の整合を保証できない。

## 結果
- 不正な遷移を一律に拒否でき、テストで網羅できる。
- 遷移条件の追加は表の変更に閉じる。
- 状態と権限の正本がサーバーに一本化される。
