## 0. 品質の方針

> このアプリで守る品質の下限と、特に力を入れる点をはじめに決めておく。

### 品質の下限（全体で満たす基準）
本アプリは、安定性・セキュリティ・保守性・性能/UX のいずれの観点でも一定の基準を満たすことを前提に設計する。どれか一つでも基準を下回らないように作る。

### 特に重視する観点：セキュリティ と 安定性
本アプリの目玉は「AWS環境へのIaC変更を、apply成功で終わらせず、変更前の危険検知・リスクに応じた承認・変更後の正常性確認・失敗時の復旧余地まで一貫して統制する」ことである。そのため **セキュリティ** と **安定性・信頼性** の2つを特に重視して設計する。

- **セキュリティ**：認可はサーバー側で強制する。他人の申請を編集・承認できない（IDOR対策）。監査ログは追記専用で改ざんできない。危険操作はサーバー側でブロックし、警告だけで通過させない。
- **安定性・信頼性**：状態遷移をサーバー側のステートマシンで強制し、飛び級や逆行を拒否する。不可逆操作は人間の承認とロールバック手順の登録を前提条件にする。変更後は「IaC適用成功」と「サービス正常性確認済み」を区別して記録する。

### 守る前提（重視する観点のために他を犠牲にしない）
- **保守性**：Controller/Service/Repository を分離し、例外処理を一元化する。AWS連携・IaC解析・リスク判定はインターフェースで抽象化し、将来 CloudFormation / Terraform / AWS SDK を無改修で足せる拡張点を残す。
- **性能・UX**：一覧はページネーション、詳細取得は N+1 を避ける。アクセシビリティに配慮し、リスクは色だけでなく文字ラベルとアイコンを併用する。

## 1. プロジェクト概要

### 1-1. 目的
AWS環境へのIaC変更（CloudFormation / Terraform）は、`apply` が成功すれば終わりではない。本アプリは「安全なクラウド変更管理」を次の形で実現する。

- 変更**前**に危険な内容を検知する
- 変更内容と影響範囲を**人間が理解しやすい形**で表示する
- **リスクに応じて承認フローを変える**
- 実行**前・後**の確認を記録する
- **誰が何を承認・実行したか監査**できる

> 設計の中心にある考え方：「AWS環境の変更は、applyが成功すれば終わりではない。変更前に危険を見つけ、必要な承認を得て、変更後に正常性を確認し、失敗時にも復旧できる状態を残す。」

### 1-2. 対象ユーザー
クラウドの運用変更に関わる4つの役割：申請者・レビュー者・実施者・管理者。運用者向けのため、派手さよりも「一目でリスクが分かる／危険な変更が埋もれない」を優先する。

### 1-3. スコープ
本アプリは**変更管理の統制と記録**を担う。AWS 連携は抽象化インターフェース（Port/Adapter）越しに行い、既定はモック、`aws` プロファイルで読み取り専用の実連携（IaC 差分・監視・インスタンス状態）に切り替える。backend からの書き込み系操作・実 apply の実行は本アプリの責務外とし、実 apply は人間の承認をはさむ（backend は実行せず結果を記録する）。

### 1-5. 差別化ポイント
必要のない仕組みを持ち込まず構成をシンプルに保つ方針で、有料版・フル機能版との違いを挙げる。

| 項目 | 有料版・フル機能版 | 本アプリ |
|---|---|---|
| AWS実行 | 実 apply・実行基盤連携 | 読み取りは実連携（IaC差分・監視・状態）、書き込み/実 apply は行わず人間ゲート＋記録 |
| リスク検知 | クラウド横断の大量ルール＋ML | 重要14ルールを説明文付きで実装 |
| 通知 | Slack/メール/PagerDuty | 画面内表示（将来拡張） |
| 範囲 | 統制＋実行＋コスト＋構成管理 | 統制・承認・監査・記録に集中 |

## 2. 前提条件と制約

- 技術スタックは指定どおり：Java / Spring Boot・React / TypeScript・PostgreSQL・JWT・RBAC・Docker Compose。スタックは大幅に変更しない。
- AWS連携は抽象化インターフェース（Port/Adapter）越しに行う。既定はモック、`aws` プロファイルで読み取り専用の実連携。
- 既存プロジェクトの構成・命名規則・依存関係を踏襲し、必要以上のリファクタリングや不要な抽象化を作らない。
- 機密情報は環境変数で扱い、コードやリポジトリに平文で残さない。

## 5. 非機能要件

### 5-1. セキュリティ（重視）
- パスワードは bcrypt（コスト10以上）で保存する。
- 認可は必ずバックエンドで判定する。ステータス遷移・承認・実行の権限をサーバー側で確認する。
- 他人の変更申請の編集 / 承認は所有者・権限の不一致で 403 / 404 にする（IDOR対策）。
- 役割は REQUESTER / REVIEWER / OPERATOR / ADMIN の4種。監査ログは追記専用で、通常ユーザーは更新・削除できない。
- 入力はエスケープして表示する。差分はプレーンテキストとして扱う。
- 永続化は JPA / プリペアドステートメント。文字列結合の SQL は使わない。
- JWT は access を短命にし refresh と組み合わせる。401 はインターセプターで一元処理する。
- 機密情報は環境変数で扱う。CORS は自分のオリジンのみ許可する。
- 危険操作（ログイン・承認・実行開始）にはレートリミットを設ける。

### 5-2. 安定性・信頼性（重視）
- 可用性 SLA は現段階では定めない（定めないことを明記する）。
- ステータス別件数などの集計は表示時に集計する。
- 変更申請の取消はソフト（CANCELLED ステータス）にし、物理削除しない。
- DB は Flyway で管理し、後方互換を保って変更する。
- 不可逆操作（本番の削除・置換）は、人間の承認とロールバック手順の登録を遷移の前提にする。
- 状態遷移は許可された遷移のみ受理する。

### 5-3. 保守性・拡張性
- すべての変更は Issue 起点・PR 経由・日本語の規約コミットで行う。
- Controller / Service / Repository を分離する。リスク判定・ポリシー・状態遷移は専用 Service に置く。
- 例外はグローバル例外ハンドラで一元処理し、API レスポンス形式を統一する。
- dev / prod のプロファイルを分ける。
- CI は lint → build → test の順で早く落とす。backend / frontend は並列。
- テストは単体（JUnit）・結合・フロント（Vitest）・可能なら E2E（Playwright）。テスト専用 DB を使う。
- AWS連携・IaC解析・リスクルールはインターフェースで抽象化し交換可能にする。API は `/api/v1/`。
- UI を伴う変更は実ブラウザで確認する。

### 5-4. 性能・UX
- API 応答はローカルで p95 < 300ms、一覧の初回表示は 1.5 秒以内を目標にする。
- 変更申請一覧・監査ログはページネーションする。
- 申請詳細（承認・チェック・監査）の取得は JOIN FETCH で N+1 を避ける。
- アクセシビリティに配慮する。リスクは色だけに依存せず、文字ラベルとアイコンを併用する。
- 実行開始・却下・ロールバックなどには確認ダイアログを出す。本番環境の変更は明確に区別して表示する。

## 6. 受入条件

- **A-1**：REQUESTER が他人の変更申請を編集できない。
- **A-2**：REQUESTER が自分の申請を承認できない（自己承認禁止）。
- **A-3**：承認済み以降の変更を申請者が編集できない。
- **A-4**：production 環境の RDS 削除がブロックされ、理由が画面と API で提示される。
- **A-5**：SSH 22 番の 0.0.0.0/0 公開がブロックされる。
- **A-6**：CRITICAL 変更で REVIEWER 2名以上の承認が必要になる。
- **A-7**：production で必須チェック未完了では実行開始（IN_PROGRESS）に遷移できない。
- **A-8**：上記いずれの操作でも監査ログが作成され、通常ユーザーが改ざんできない。
- **A-9**：状態遷移が許可された経路以外を拒否する（DRAFT から直接 COMPLETED 不可など）。
- **A-10**：実施後ヘルスチェックで「IaC適用成功」と「サービス正常性確認済み」が区別される。

## 7. 用語集（本アプリ内）

| 用語 | 意味 |
|---|---|
| 変更申請（Change Request） | AWS環境への1つの変更単位。状態を持つ |
| リスク判定 | 差分・項目から危険度（LOW/MEDIUM/HIGH/CRITICAL）と検知ルールを導く処理 |
| ポリシー | 環境×条件に対する統制効果（ブロック・追加承認必須・理由必須など） |
| 実施前チェック | 実行前に完了させる確認項目。本番は必須項目あり |
| 実施後ヘルスチェック | 変更後の正常性確認記録。IaC適用成功とサービス正常性を区別 |
| 監査ログ | 操作の追記専用記録。編集・削除不可 |

## 10. リスク・前提（R-* 形式）

| ID | リスク | 対策 |
|---|---|---|
| R-01 | リスク判定が文字列一致頼みで誤検知 / 見逃し | 差分の正規化＋登録項目の併用。ルールを単体テストで検証 |
| R-02 | フロントだけの認可制御で抜ける | 認可はサーバー強制。認可マトリクスをテスト |
| R-03 | 状態遷移の不正遷移 | 許可遷移表＋ガードをサーバーで強制し、不正パターンをテスト |
| R-04 | 監査ログ改ざん | 追記専用・更新 / 削除をアプリ層で禁止 |
| R-05 | AWS連携を密結合し将来差替不能 | Port/Adapter で抽象化。既定モック・aws で読み取り専用の実連携 |
| R-06 | 過剰設計・未使用コード増殖 | MVP スコープ厳守。既存命名 / 構成を踏襲 |
