# 開発前チェックと既存アプリの教訓（CloudOps Change Guard）

> 実装セッションに入る前に、過去アプリ（term-board / review-board / aws-zukan）の一次知見と踏んだつまずきを取り込み、**同じトラブルを未然に防ぐ**ための前提整理。各項目は「教訓 → 本アプリでの対策」で示す。

---

## 1. 既存アプリの教訓 → 本アプリの対策

### 1-1. term-board（フロント先行 SPA）からの学び
| 教訓 | 本アプリでの対策 |
|---|---|
| **版数整合**：実機ランタイムと build toolchain を一致させ、奇数版（非LTS）を避ける | Java 25(LTS) / Gradle 9.1 / Spring Boot 3.5.x / Node LTS で統一（[技術スタック.md] 版数整合） |
| **api/ 隔離 = 差し替え点を1箇所に**（localStorage→http を無改修移行） | AWS連携を Port/Adapter で隔離（[AWS・IaC連携方針.md]）。Service は SDK を直接呼ばない |
| **graceful degradation**：壊れても機能を止めない（破損 localStorage は進捗のみ初期化し継続） | 破損 diff でパーサを落とさず `UNKNOWN`/空 findings で安全側に倒す。外部取得は null ガード（[テスト計画書.md] §4） |
| **テスト計画は"正直に現状を記す生きた文書"**（未導入は未導入と明記） | 各 Phase で実態を更新。テスト緑≠UI 動作（M-13） |
| **CI に機密/依存脆弱性スキャンを常設**（gitleaks/npm audit/CodeQL・High/Critical 0件） | Phase 7 CI に gitleaks＋依存スキャン＋静的解析を常設（[テスト計画書.md] §6） |
| **Tier 移行ログ運用**：Tier 変更時は記録→防御適用→CLAUDE.md 更新 | 初期 Tier 確定済み（[Tier移行ログ.md]）。AWS 採用は Phase 6 で判断 |
| **採用しない技術と理由を明記**（過剰機能で短所を作らない） | Ansible 保留等を明記（[技術スタック.md]） |

### 1-2. aws-zukan（AWS図鑑）からの学び
| 教訓（横断ルール/サービス知見） | 本アプリでの対策 |
|---|---|
| `deploy-audit`：apply 前点検（機密grep・環境固有値の外部化・無料枠・版数突合・**plan 差分の削除/置換目視**・削除保護） | リスク判定＋ポリシー＋実施前チェックで**人手でなく仕組みで強制**（[リスク判定ルール.md] §6） |
| `iam-templates`：役割で権限を分ける（読取/運用/デプロイ/監査） | 4ロールの職務分掌（[権限マトリクス.md]）。自己承認禁止 |
| `network`/`securitygroup`：SSH(22) を 0.0.0.0/0 で開けない・許可元は SG 優先 | SSH_22_OPEN_WORLD / SG_OPEN_WORLD を BLOCK |
| `backup`：自動バックアップ・別アカウント複製・復元テスト | RDS 削除を高リスク化、ロールバック手順を遷移前提に |
| `logging`：CloudTrail/Config の改ざんされにくい保管 | CloudTrail/Config 停止・削除を高リスク化。監査ログは追記専用 |
| service `cloudformation`：失敗時の自動ロールバックで**成功リソースも削除**。原因は events を見る | 「IaC適用成功≠サービス正常」を別管理。実施後ヘルスチェックに適用結果＋events 参照を残す |
| service `ec2`：EIP/パブリックIPv4 は課金。`ssm`：Session Manager で踏み台レス | Phase 6 コスト注意に反映。Ansible は SSM 経由前提 |

### 1-3. CLAUDE.md 事故事例（§13 Terraform 本番全削除）からの学び
| 教訓 | 本アプリでの対策 |
|---|---|
| AI/自動化が destroy を無制限に実行できる構成は危険 | 不可逆操作は人間承認＋復旧手順を遷移前提に（S-6）。TF_DESTROY_BULK を CRITICAL/BLOCK 候補に |
| バックアップは別アカウント（同一削除で消えない） | 復旧設計の前提として明記（実装は対象外だがチェック項目化） |

## 2.5 既存アプリ内容の網羅調査で得た追加の参考素材

> aws-zukan（112サービス／24構成パターン／4 ADR／共通ルール／不適合構成）と term-board（277語＋面接データ）の**中身を網羅調査**して得た、cloudops 開発に効く追加素材。一次ファイルで裏取り済み。

### aws-zukan から
- **patterns の `antiPatterns`（stack/why/source）**：構成パターンごとに「やってはいけない理由＋出典」が構造化されている。cloudops のリスクルール文面・将来 Phase 6 構成の落とし穴チェックに転用可（例：pat-alb-frontend＝「SSH22公開」「ALB を 2AZ 未満で作成不可」）。
- **ADR-0003「派生ビューはルールベース（文章生成に頼らない）」／ADR-0004「断定せず前提条件つきで示す」**：cloudops のリスク説明方針に反映済み（[リスク判定ルール.md] §7）。**cloudops 自身も `docs/ADR/` を設け、主要設計判断（状態機械・認可・リスクルール）を 1決定1ファイルで記録**する（設計判断を ADR で残すのは保守性のよい習慣）。
- **ADR-0002「8軸評価（Well-Architected＋独自3軸）」**：リスクの影響を単一値でなく複数軸（セキュリティ/可用性/コスト等）で見る発想。将来のリスク影響度の多軸表示の参考。
- **costGotcha（停止しても課金）**：EC2 停止中の EBS／未割当 Elastic IP・パブリック IPv4／NAT／RDS storage・backup は課金継続。→ 実施前チェック（IMPACT/コスト）と Phase 6 コスト注意に反映。
- **Phase 6 構成パターンの実体**：pat-alb-frontend / pat-rds-access / pat-tf-state / pat-ssm-session-deploy / pat-security-monitoring / pat-web-3tier を、stackRoles・realWorldNotes 付きでそのまま設計の下敷きにできる。

### term-board から
- **説明の5層モデル**（正確な定義／平易版／場面／深掘り／一言）：cloudops のリスク説明文の書き方に反映済み（[リスク判定ルール.md] §7-3、[画面設計書.md]）。
- **ドメイン用語のすぐ使える説明**：認可/JWT/IDOR/CORS/SQLi/XSS/トランザクション/排他制御/ステートマシン/CI-CD/単体・E2Eテスト等の「意味＋平易版＋場面」。cloudops のツールチップ・監査説明・社内資料に転用可。
- **平易な言葉ファースト＋段階的開示**：運用者向け UI 原則として反映済み（[画面設計書.md]）。
- **api/ 隔離＝差し替え点を1箇所に**：cloudops の Port/Adapter 方針（[AWS・IaC連携方針.md]）を補強。

## 3. 実装セッション冒頭の推奨順序

1. CLAUDE.md 個別化（#4）＋ Tier/ポート確認。
2. pre-commit フック導入（#9）→ 禁止語ブロックを確認（ポカヨケ先行）。
3. application*.yml（#5）＋ .env.example（#8）。
4. Flyway V1__init.sql（#6）＝ ER図の12テーブル。
5. Phase 2 実装（認証→RBAC→CRUD→状態遷移→監査→コメント→JUnit）。
6. docker-compose.yml（#7）でローカル起動・動作確認。
