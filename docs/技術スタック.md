|---|---|---|

## 拡張点（将来の差し替えを無改修で）

- `IacDiffParser`：CloudFormation / Terraform の差分パーサ（IaC種別ごとに実装）。
- `RiskRule` / `RiskEngine`：リスクルール集合。1ルール1クラス。
- `PolicyEvaluator`：ポリシー効果の確定。
- `AwsChangeExecutor`（将来）：実 apply 実行アダプタ。MVP はモック実装。

## ディレクトリ方針（既存アプリ群の慣習を踏襲）

```
cloudops-change-guard/
├── docs/                 ← 本設計群
├── backend/              ← Spring Boot（domain/ で機能分割：auth, changerequest, risk, policy, approval, audit, dashboard）
├── frontend/             ← React + Vite + TS（api/ 隔離）
├── docker-compose.yml
└── README.md
```

> 既存（review-board 等）の Controller/Service/Repository 分割・`api/` 隔離・命名規則に合わせ、不要な抽象化は作らない。

## 版数整合の方針（既存アプリの教訓）

term-board の教訓「実機ランタイムと build toolchain を一致させ、奇数版（非LTS）を避ける」を取り入れる（aws-zukan `deploy-audit`④ とも一致）。

| 対象 | 採用版 | 理由 |
|---|---|---|
| Java | **25 (LTS)** | 他アプリ（review-board 等）と統一。Gradle toolchain で自動取得 |
| Gradle | 9.1 | wrapper 固定で再現性 |
| Spring Boot | 3.5.x | 他アプリと統一。依存脆弱性は版上書きで High/Critical 0 件 |
| Node | LTS（偶数版） | 非LTS（奇数版）を避けビルド再現性を確保 |

## 採用しない技術と理由（過剰機能を持たない＝攻撃面・複雑性を増やさない）

| 技術 | 不採用/保留の理由 | 将来 |
|---|---|---|
| Ansible | MVP に OS 内部設定の自動化要件がない。Terraform（資源）＋Docker（実行環境）＋SSM（安全な運用操作）で充足 | 条件該当時に SSM 経由で再検討（[AWS・IaC連携方針.md] §4） |
| 実 AWS SDK 呼び出し | MVP は Mock/抽象化。密結合を避け Port/Adapter で隔離 | Phase 5 以降で Adapter 実装 |
| 状態管理ライブラリ（Redux 等） | 規模に対し過剰。Context＋フックで足りる | 必要時に検討 |
| メッセージング/通知基盤 | MVP は画面内表示で充足 | 将来 SNS/メール |

> 「採用しない」を明記するのは、不要な仕組みを持ち込んで短所（攻撃面・保守コスト）を作らないため。禁止事項「大量の未使用コードや不要な抽象化を作らない」と一致。
