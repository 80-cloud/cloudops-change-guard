# 権限マトリクス（CloudOps Change Guard）

> 認可は**必ずバックエンドで強制**する。フロントの表示制御は補助に過ぎない。所有者検証（IDOR対策）を併用する。

## 1. ロール定義

| ロール | 役割 |
|---|---|
| REQUESTER | 変更申請の作成。自分の申請の閲覧。下書き(DRAFT)/差し戻し(RETURNED)状態の自分の申請のみ編集 |
| REVIEWER | 申請内容の確認。承認・却下・差し戻し。自分が申請した変更は承認不可 |
| OPERATOR | 承認済み変更の実施開始。実施前/後チェック記録。実施失敗・ロールバック記録 |
| ADMIN | 全申請の閲覧。ポリシー設定の閲覧・管理。監査ログ閲覧 |

> MVP は 1ユーザー1ロール（`users.role` enum）。将来の複数ロールは `roles` / `user_roles` 中間表で拡張可能（今回は対象外）。

> **設計根拠（aws-zukan `iam-templates`：権限分離の型）**：AWS図鑑の横断ルール「役割ごとに権限を分ける（読み取り専用＝閲覧/監査・運用者＝再起動/スケール・デプロイ担当＝更新/リリース・監査担当＝ログ/証跡参照）。人ではなく役割(ロール)で最小権限を付与する」を、本アプリの4ロールに対応づけている：
> - **デプロイ担当（更新/リリースの起案・査閲）** → REQUESTER（申請）＋ REVIEWER（承認）
> - **運用者（実行・再起動・スケール）** → OPERATOR
> - **監査担当（ログ/証跡の参照）・読み取り専用** → ADMIN（監査ログ・全申請閲覧）
> これは「変更を起こす人」と「承認する人」と「実行する人」と「監査する人」を分ける**職務分掌（SoD）**であり、自己承認禁止（A-2）もこの分離の一部。

## 2. 操作 × ロール マトリクス

凡例：✅=可 / ⛔=不可 / 👤=自分のリソースのみ / ＊=条件付き

| 操作 | REQUESTER | REVIEWER | OPERATOR | ADMIN |
|---|---|---|---|---|
| 変更申請を作成 | ✅ | ⛔ | ⛔ | ⛔ |
| 一覧/詳細を閲覧 | 👤(自分) | ✅ | ✅ | ✅ |
| 変更申請を編集 | 👤＊(DRAFT/RETURNED) | ⛔ | ⛔ | ⛔ |
| 提出(submit) | 👤 | ⛔ | ⛔ | ⛔ |
| 取消(cancel) | 👤 | ⛔ | ＊(実施系) | ✅ |
| レビュー着手(review-start) | ⛔ | ✅ | ⛔ | ⛔ |
| 承認(approve) | ⛔ | ✅＊(自己承認不可) | ⛔ | ⛔ |
| 却下(reject) | ⛔ | ✅ | ⛔ | ⛔ |
| 差し戻し(return) | ⛔ | ✅ | ⛔ | ⛔ |
| 予定設定(schedule) | ⛔ | ⛔ | ✅ | ⛔ |
| 実施開始(start) | ⛔ | ⛔ | ✅＊(前チェック/承認/非BLOCK) | ⛔ |
| 実施完了/失敗(complete/fail) | ⛔ | ⛔ | ✅ | ⛔ |
| ロールバック(rollback) | ⛔ | ⛔ | ✅ | ⛔ |
| 実施前チェック更新 | ⛔ | ⛔ | ✅ | ⛔ |
| 実施後ヘルスチェック更新 | ⛔ | ⛔ | ✅ | ⛔ |
| コメント投稿 | ✅ | ✅ | ✅ | ✅ |
| 監査ログ閲覧（当該申請） | 👤 | ✅ | ✅ | ✅ |
| 監査ログ閲覧（全体） | ⛔ | ⛔ | ⛔ | ✅ |
| 監査ログ編集/削除 | ⛔ | ⛔ | ⛔ | ⛔（誰も不可・追記専用） |

| ポリシー管理 | ⛔ | ⛔ | ⛔ | ✅ |

## 3. 認可の実装方針

- **メソッドレベル認可**：Spring Security の `@PreAuthorize("hasRole('REVIEWER')")` 等でロールを強制。
- **所有者検証（IDOR対策）**：編集/閲覧時に `changeRequest.requesterId == authUser.id` を Service で検証。不一致は 404（存在秘匿）または 403。
- **自己承認禁止**：approve 時に `reviewerId != requesterId` を検証（受入 A-2）。
- **監査ログの不可変性**：監査ログに UPDATE/DELETE の API・Repository メソッドを設けない（受入 A-8）。
- フロントは同じ判定をUI出し分けに使うが、**サーバーが最終判定**。テストで全マトリクスを保証（[テスト計画書.md]）。
