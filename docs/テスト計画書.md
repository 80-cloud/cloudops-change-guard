# テスト計画書（CloudOps Change Guard）

> テストはピラミッド型（単体＝ロジック / 結合＝システム接続 / E2E＝ユーザー体験）。テスト専用 DB で本番に触れない。カバレッジは目安（85%・重要経路100%）。
>
> **本書は「現状を正直に記す生きた文書」**（term-board の運用に倣う）。未実装のテストは「未実装」と明記し、各 Phase 完了時に実態を更新する。テスト緑 ≠ UI が動く（M-13）。

## 1. レイヤと道具

| レイヤ | 道具 | 主対象 |
|---|---|---|
| 単体（backend） | JUnit / Mockito | リスクルール・ポリシー評価・状態機械・認可 |
| 結合（backend） | Spring Boot Test + Testcontainers(PostgreSQL) | API・永続化・監査ログ追記 |
| 単体/コンポーネント（frontend） | Vitest + Testing Library | フォーム・リスク表示・ガード表示 |
| E2E（可能なら） | Playwright | 申請→承認→実行→監査の主要動線 |

## 2. 必ずテストするケース（受入条件と対応）

| # | テスト内容 | 受入 | レイヤ |
|---|---|---|---|
| T-1 | REQUESTER が他人の変更申請を編集できない（403/404） | A-1 | 結合 |
| T-2 | REQUESTER が自分の申請を承認できない（自己承認禁止） | A-2 | 単体/結合 |
| T-3 | 承認済み変更を申請者が編集できない | A-3 | 結合 |
| T-4 | production の RDS 削除がブロックされる（理由提示） | A-4 | 単体/結合 |
| T-5 | SSH 22 番の全世界公開がブロックされる | A-5 | 単体/結合 |
| T-6 | CRITICAL 変更で REVIEWER 2名以上の承認が必要 | A-6 | 結合 |
| T-7 | 必須チェック未完了では IN_PROGRESS に遷移できない（本番） | A-7 | 結合 |
| T-8 | 各操作で監査ログが作成され改ざんできない | A-8 | 結合 |
| T-9 | 不正な状態遷移（DRAFT→COMPLETED 等）が拒否される | A-9 | 単体/結合 |
| T-10 | IaC適用成功とサービス正常性確認が区別される | A-10 | 結合 |

## 3. リスク判定の単体テスト方針

- 各 `RiskRule` に「該当差分で検知 / 非該当で不検知 / 環境で level 変化」の3系統。
- サンプル差分フィクスチャ（Terraform plan / CloudFormation change set 風）を resources に置き回帰テスト化。
- 誤検知・見逃し（R-01）を検出する境界値ケースを含める。

## 4. 堅牢化テスト（既存アプリの教訓＝壊れても止めない）

> term-board の「localStorage 破損でも進捗のみ初期化し出題は継続（graceful degradation）」を、本アプリでは**外部 IaC データ・差分パーサの堅牢化**として取り入れる。

- **破損 diff でパーサが落ちない**：不正・空・巨大・想定外形式の差分を渡しても 500 にせず、`UNKNOWN` 判定や空 findings で**安全側に倒す**（誤って「危険なし」と誤判定しない＝検知不能は UNKNOWN として扱い、人手レビューへ）。
- **外部取得結果の null ガード**（母 §3-3）：監視 Adapter / IaC Provider の戻りが null/空でも例外で止めない。
- **Mock Adapter で AWS 非接続でも全機能が動作**する（Phase 5 で検証）。

## 5. テスト独立性・flaky 対策

- 結合テストは Testcontainers で毎回まっさらな DB（M-8）。本番・共有 DB に触れない。
- 固定 sleep でなく条件待ち（M-10）。E2E は失敗時スクショ/トレース保存。

## 6. CI と品質ゲート（既存アプリの教訓を常設）

> term-board / review-board の CI 運用に倣い、**機密スキャン・依存脆弱性スキャンを最初から常設**する（aws-zukan `deploy-audit`① と一致）。

- **Fail Fast**：lint → build → test（M-6）。backend / frontend 並列。
- **機密スキャン**：gitleaks を PR/main で常設（平文の鍵・トークンの混入を機械検知）。
- **依存脆弱性スキャン**：backend は Gradle 依存（Trivy 等）、frontend は `npm audit`。**High/Critical 0 件**を維持（検出時はバージョン上書きで解消）。
- **静的解析**：可能なら CodeQL。
- **Terraform（Phase 6 以降）**：CI では `fmt -check` / `validate` / `plan` のみ。**本番 apply は実行しない**。plan は Artifact 保存（将来 CloudOps Change Guard へ取り込む布石）。
- 主要受入（T-1〜T-10）が緑でなければマージ不可を運用ルールにする。

## 7. 版数整合（build / runtime 一致＝term-board の教訓）

ビルドツールと実行ランタイムの版数を一致させ、奇数版（非LTS）を避ける。Java 25(LTS) / Gradle 9.1 / Spring Boot 3.5.x / Node LTS を他アプリと統一（aws-zukan `deploy-audit`④ と一致）。
