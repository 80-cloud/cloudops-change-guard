# インフラ構成（CloudOps Change Guard）

> MVP はローカル Docker Compose で完結。実 AWS は将来候補（要件定義書のスコープ参照）。

## 1. ローカル構成（Docker Compose）

```
┌─────────────┐   :5177   ┌──────────────┐   :8084   ┌──────────────┐   :5436
│  frontend   │ ───────▶ │   backend    │ ───────▶ │  PostgreSQL  │
│ React/Vite  │  (proxy) │ Spring Boot  │  (JPA)   │  (Docker)    │
└─────────────┘           └──────────────┘           └──────────────┘
        AWS連携は backend 内の抽象化インターフェース（MVPはモック実装）
```

| サービス | ポート | 備考 |
|---|---|---|
| frontend（Vite） | 5177 | `/api` を backend へプロキシ |
| backend（Spring Boot） | 8084 | CORS は 5177 のみ許可 |
| PostgreSQL | 5436 | Flyway で初期化 |

> ポートは既存アプリ（task-board 5173/8080/5432・review-board 5175/8082/5434）と衝突しない連番。CORS・プロキシは上記前提で固定。

## 2. 環境変数（機密はコミットしない）

| 変数 | 用途 |
|---|---|
| `DB_URL` / `DB_USER` / `DB_PASSWORD` | DB 接続 |
| `JWT_SECRET` | JWT 署名鍵 |
| `JWT_ACCESS_TTL` / `JWT_REFRESH_TTL` | トークン有効期限 |
| `BOOTSTRAP_ADMIN_*` | 初期 ADMIN 作成 |

`.env.example` を用意し、実値は `.env`（git 追跡外）で管理。

## 3. 将来の AWS 展開（候補・MVP対象外）

- 実 apply 実行は `AwsChangeExecutor` 実装を追加（CloudFormation / Terraform / AWS SDK）。
- デプロイ Tier は着手時に [インフラ選択ガイド.md](./インフラ選択ガイド.md) で確定（AWS を既定にしない）。
- 本番 DB・破壊的操作は人間承認をはさむ多層防御を前提とする。

### 3-1. infra/ のデプロイ雛形（Terraform・apply なし）

`infra/` に AWS 上での構成（VPC / サブネット / VPC フローログ / KMS 鍵 / RDS PostgreSQL / EC2 / IAM ロール / 予算アラート）を Terraform で記述している。現時点では雛形で、実際の terraform apply は行わない。認証情報なしで実行できる静的検査（fmt / validate / tflint / tfsec）をローカルと CI（`.github/workflows/infra-check.yml`）の関門にする。重要リソース（RDS）には prevent_destroy と deletion_protection を設定し、誤削除を防ぐ。
