# インフラ構成（CloudOps Change Guard）

> 開発はローカル Docker Compose、公開環境は AWS（EC2 + nginx + RDS）に常時デプロイしている。

## 1. ローカル構成（Docker Compose）

```
┌─────────────┐   :5177   ┌──────────────┐   :8084   ┌──────────────┐   :5436
│  frontend   │ ───────▶ │   backend    │ ───────▶ │  PostgreSQL  │
│ React/Vite  │  (proxy) │ Spring Boot  │  (JPA)   │  (Docker)    │
└─────────────┘           └──────────────┘           └──────────────┘
        AWS連携は backend 内の抽象化インターフェース（既定はモック、aws プロファイルで読み取り専用の実連携）
```

| サービス | ポート | 備考 |
|---|---|---|
| frontend（Vite） | 5177 | `/api` を backend へプロキシ |
| backend（Spring Boot） | 8084 | CORS は 5177 のみ許可 |
| PostgreSQL | 5436 | Flyway で初期化 |

> ポートは既存アプリ（task-board 5173/8080/5432・review-board 5175/8082/5434）と衝突しない連番。CORS・プロキシは上記前提で固定。

## 2. 環境変数（機密はコミットしない）

| 変数 | 用途 |
|---|---|
| `DB_URL` / `DB_USER` / `DB_PASSWORD` | DB 接続 |
| `JWT_SECRET` | JWT 署名鍵 |
| `JWT_ACCESS_TTL` / `JWT_REFRESH_TTL` | トークン有効期限 |
| `BOOTSTRAP_ADMIN_*` | 初期 ADMIN 作成 |

`.env.example` を用意し、実値は `.env`（git 追跡外）で管理。

## 3. 公開 AWS 構成

```
Internet ─(EIP)─▶ EC2（public・IMDSv2）
                   └ nginx（React 配信＋ /api → :8084）
                        └ Spring Boot ─▶ RDS PostgreSQL（private）
```

- ネットワーク：public サブネットの EC2（Elastic IP・IMDSv2 必須）、RDS は private サブネット。app のセキュリティグループから RDS へ 5432 のみ許可（ingress/egress とも明示）。
- 配信：nginx が React ビルドを配信し、`/api` を backend（:8084）へリバースプロキシ。TLS は Let's Encrypt（certbot 自動更新）。
- 暗号化・保護：KMS 暗号化、RDS は prevent_destroy と deletion_protection。予算アラート（Budgets）で無料枠を監視。
- 実行系：EC2 は t3.micro、RDS は db.t3.micro（無料枠）。実 apply は人間の承認をはさんで実行し、backend からは行わない（記録に限定）。
- デプロイ Tier は [インフラ選択ガイド.md](./インフラ選択ガイド.md) で確定（[要件定義書.md](./要件定義書.md) のスコープ参照）。

### 3-1. infra/ の Terraform 構成

`infra/` に上記構成（VPC / サブネット / VPC フローログ / KMS 鍵 / RDS PostgreSQL / EC2 / IAM ロール / 予算アラート / 成果物 S3）を Terraform で記述する。apply は人間が承認して実行し、backend からは行わない。認証情報を持たない静的検査（fmt / validate / tflint / tfsec）をローカルと CI（`.github/workflows/infra-check.yml`）の関門にする。重要リソース（RDS）には prevent_destroy と deletion_protection を設定し、誤削除を防ぐ。

### 3-2. デプロイ（CD）

GitHub Actions で成果物をビルドして S3 に置き（不変 SHA）、SSM Run Command で EC2 へ配備する（`.github/workflows/cloudops-cd-build.yml` / `cloudops-cd-deploy.yml`）。配備は手動トリガの承認ゲートを通す。OIDC の鍵レス認証で、長期の認証情報を CI に置かない。
