# ポリシー一覧・ポリシーエンジン（CloudOps Change Guard）

> ポリシーは `環境スコープ × 条件 → 効果` のルール集合。**サーバー側の状態遷移ガードで評価**し、BLOCK 該当は SUBMIT/APPROVE/START を拒否する。**警告だけで通過させない**。違反時は「なぜ実行できないか／何が必要か」を画面と API レスポンスで明示する。

## 1. 効果（effect）の種類

| effect | 意味 | 遷移への影響 |
|---|---|---|
| BLOCK | 実行不可 | SUBMIT/APPROVE/START を拒否（409）。理由を返す |
| REQUIRE_DUAL_APPROVAL | 二重承認必須 | REVIEWER 2名以上の APPROVED まで APPROVED にしない |
| REQUIRE_ADDITIONAL_APPROVAL | 追加承認必須 | 通常段数＋1名 |
| REQUIRE_REASON | 理由入力必須 | change_reason 未入力なら遷移拒否 |
| REQUIRE_MAINTENANCE_WINDOW | メンテ時間登録必須 | scheduled_at（メンテ枠）未登録なら遷移拒否 |

## 2. 初期ポリシー（MVP 投入データ）

| code | name | scope | 条件 | effect |
|---|---|---|---|---|
| PROD_RDS_DELETE_BLOCK | 本番RDS削除のブロック | production | RDS delete/replace 検知 | BLOCK |
| PROD_S3_DELETE_BLOCK | 本番S3削除のブロック | production | S3 bucket delete 検知 | BLOCK |
| SG_OPEN_WORLD_BLOCK | SG全世界公開のブロック | ALL | SG 0.0.0.0/0 公開検知 | BLOCK |
| IAM_ADMIN_ADDITIONAL_APPROVAL | IAM管理者権限の追加承認 | ALL | IAM AdministratorAccess 付与検知 | REQUIRE_ADDITIONAL_APPROVAL |
| CW_ALARM_DELETE_REASON | CloudWatch Alarm削除の理由必須 | ALL | Alarm delete 検知 | REQUIRE_REASON |
| EC2_REPLACE_WINDOW | EC2置換のメンテ時間必須 | production | EC2 replace 検知 | REQUIRE_MAINTENANCE_WINDOW |
| CRITICAL_DUAL_APPROVAL | CRITICAL変更の二重承認 | ALL | risk_level=CRITICAL | REQUIRE_DUAL_APPROVAL |

> SSH22全世界公開は SG_OPEN_WORLD_BLOCK（および SSH_22 検知）で BLOCK 対象（受入 A-5）。本番RDS削除は PROD_RDS_DELETE_BLOCK で BLOCK（受入 A-4）。CRITICAL は CRITICAL_DUAL_APPROVAL で二重承認（受入 A-6）。

## 3. 評価タイミングと責務分離

- **リスク判定（[リスク判定ルール.md]）** … 何が危険かを説明する（findings）。
- **ポリシーエンジン** … findings＋環境を入力に、統制効果（BLOCK等）を確定し、遷移ガードへ渡す。
- 評価点：
  - **SUBMIT 時**：BLOCK 該当 → 提出拒否＋違反記録。
  - **APPROVE 時**：REQUIRE_DUAL/ADDITIONAL_APPROVAL の段数充足を確認。
  - **START 時**：REQUIRE_MAINTENANCE_WINDOW・REQUIRE_REASON 等の前提を再確認（多層防御）。

## 4. 違反レスポンス例

```json
{
  "error": {
    "code": "POLICY_BLOCKED",
    "message": "本番環境のRDS削除はポリシーにより実行できません",
    "details": [{
      "policy": "PROD_RDS_DELETE_BLOCK",
      "effect": "BLOCK",
      "reason": "本番DBの削除はデータ消失と全断に直結するため、本アプリでは実行をブロックします。削除が必要な場合は別途承認プロセスと事前バックアップ確認が必要です。"
    }]
  }
}
```

## 4.5 ポリシー実データ（実装セッションで使用）

初期ポリシー8件を `backend/src/main/resources/data/policy-rules.json` に作成済み（§2の7件＋SSH22の明示BLOCK）。スキーマ：
```
code, name, description, environmentScope(development/staging/production/ALL),
effect(BLOCK/REQUIRE_DUAL_APPROVAL/REQUIRE_ADDITIONAL_APPROVAL/REQUIRE_REASON/REQUIRE_MAINTENANCE_WINDOW),
appliesToRuleCodes[] | appliesToRiskLevels[], message(画面/APIに返す理由文), enabled
```
> `PolicyEvaluator` がこのデータと検知結果(findings)を突き合わせ、効果を確定する。MVP は投入＋一覧表示。

## 5. 実装方針

- `PolicyRule` を DB（policy_rules）に保持し、条件判定は `PolicyEvaluator`（コード側、ルールコード対応）で行う（M-3/M-11）。
- 違反は policy_violations に記録し、監査ログ action_type=POLICY_VIOLATION を残す。
- MVP は初期データ投入＋一覧表示（ADMIN）。enabled フラグで有効/無効のみ切替。フル CRUD は将来。
- **BLOCK を警告で通過させない**：遷移サービスが BLOCK を検知したら例外で停止（禁止事項の機械的担保）。テストで A-4/A-5 を保証。
