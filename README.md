# CloudOps Change Guard

AWS 環境への IaC 変更（CloudFormation / Terraform）を**安全に統制する**変更管理アプリ。`apply` の成功で終わらせず、**変更前の危険検知 → リスクに応じた承認 → 変更後の正常性確認 → 失敗時の復旧余地**までを一貫して扱います。

**ライブデモ**：https://cloudops-guard.duckdns.org/ （登録不要・デモ管理者ボタンで、リスク判定〜承認〜完了ゲートまで操作できます）

## なぜ必要か

クラウドの構成変更は、一度の操作で本番を壊しかねません。それなのに変更管理は形だけの申請フローになりがちで、実際の危険（削除・置換・本番への影響）を止められないことが多い。個人開発で AWS を触るほど、「機能を足すこと」より **「変更を安全に回す仕組み」** のほうが難しく、価値があると感じました。

そこで、申請フォームを埋めるのではなく、**実際の IaC 差分から危険度を判定し、リスクに応じて承認の重さを変え、変更後の正常性まで確認して初めて「完了」とする**変更管理アプリを作りました。

## 主要機能

- **変更申請**：IaC 種別（Terraform / CloudFormation）・対象・差分・切り戻し手順を登録。
- **リスク判定**：差分を「リソース型 × アクション」に正規化し、ルールで危険度（低〜重大）を自動判定。
- **ポリシーと承認フロー**：リスクに応じて必要な承認段数が変わる。条件を満たさない状態遷移は拒否。
- **状態遷移（ステートマシン）**：申請 → 査閲 → 承認 → 実施 → 完了を、決められた道筋でのみ進める。
- **完了ゲート**：適用の成否とサービス正常性を別々に記録し、両方そろうまで完了にしない。
- **監査ログ**：誰がいつ何をしたかを追記専用で記録（後から書き換えない）。
- **ダッシュボード**：状態別の件数・高リスク申請・承認待ち・操作履歴を一覧。
- **実 AWS 連携（read-only）**：実際の plan や監視状態を参照して、変更後の正常性を判定。
- **デモログイン**：登録不要で試せるデモ管理者アカウント。

## 何が違うか

「変更管理」や「`plan` の承認」だけを行う仕組みは多いですが、本アプリは **「`apply` が通ったか」ではなく「安全に変わり、正常だと確認でき、失敗しても戻せる状態か」** までを 1 つのライフサイクルで扱う点が違います。

| よくあるやり方 | 本アプリの違い |
|---|---|
| フォーム記入ベースの変更申請 | 実際の IaC 差分からリスクを**機械判定**（記入内容でなく差分を見る） |
| `plan` の承認・ポリシーチェックで完了 | 適用の成否とサービス正常性を**分離**し、両方そろうまで完了にしない |
| 一律の承認フロー | **リスクに応じて承認段数が変わる**。条件を満たさない遷移は拒否 |
| 文字列一致でのリスク検出 | **データ駆動のルール（JSON）**で判定し、ルールを足すだけで拡張 |

## 強み

- **完了ゲート**：適用の成否とサービス正常性を分けて持ち、両方そろうまで完了にしない。
- **データ駆動のリスク判定**：文字列一致ではなくルールで判定するため、ルールを足すだけで拡張できる。
- **fail-closed な承認**：危険な変更ほど承認を増やし、不正な状態遷移はサーバーで拒否（409）。認可は必ずバックエンドで検証。
- **誤操作の歯止め（IaC 安全レール）**：IMDSv2 必須・KMS 暗号化・prevent_destroy・コスト監視・認証情報を持たない静的 CI（fmt / validate / tflint / tfsec）。
- **常時稼働と自動デプロイ**：AWS 上でライブ稼働（EC2 ＋ RDS ＋ 正規 TLS）。GitHub Actions でビルド→配信→デプロイを自動化し、昇格は人の承認ゲートを挟む。

## 主要な技術的リスクと対策

失敗しやすい所を先に洗い出し、設計で塞いでいます。

| リスク | 対策 |
|---|---|
| リスク判定が文字列一致頼みで誤検知・見逃し | 差分を正規化し構造化項目で判定。ルールは単体テストで回帰を担保 |
| フロントだけの認可で他人の申請を操作できる（IDOR） | 認可はサーバーで検証し所有者もチェック。認可のテスト群を用意 |
| 状態遷移の不正・競合でデータ不整合 | ステートマシン＋楽観ロック。不正な遷移は全件テスト |
| 監査ログの改ざんで追跡不能 | 追記専用にし、更新・削除の API を設けない |
| 「IaC 適用成功＝完了」と誤認して異常を見落とす | 適用成功とサービス正常性を別管理し、両方そろうまで完了にしない |

## ロール（認可）

| ロール | できること |
|---|---|
| REQUESTER | 変更申請の作成・自分の申請の閲覧/編集（承認済みは編集不可） |
| REVIEWER | レビュー・承認・却下・差し戻し |
| OPERATOR | 承認済み変更の実行・実施前後チェック記録・ロールバック記録 |
| ADMIN | 全申請閲覧・ポリシー管理・監査ログ閲覧 |

## AI の活用方針

設計・実装・レビュー・ドキュメント作成の各工程で、AI コーディングツールを活用しています。使い方の方針を明確にしておきます。

- **設計判断と最終責任は自分が持つ**：AI の提案はそのまま採用せず、意図と根拠を理解したうえでレビューして取り込んでいます。
- **動作は必ず自分で検証する**：テスト・実機確認・CI のグリーンを確認してからマージしています。
- **道具として使いこなす**：AI を開発を速める道具と位置づけ、品質と成果物の責任は自分にあります。
